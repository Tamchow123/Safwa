# Safwa — Implementation Phases

Status: planning baseline (Architecture Plan v4, approved 2026-07-14).

23 phases (0–22), each small enough to hand to one Claude Code session.
**Do not start a phase until the previous phase's testing checkpoint passes
and its demonstration has been shown.** Every phase ends with a commit; the
repo must be green (`typecheck · lint · test`) at every phase boundary.

Milestones:

| Milestone                        | After phase | Meaning                                                          |
| -------------------------------- | ----------- | ---------------------------------------------------------------- |
| 🏁 **Guest Alpha**               | 12          | a guest has the full local learning experience — **not** the MVP |
| 🏁 **Core MVP**                  | 17          | accounts, server-backed progress, cross-device sync, guest merge |
| 🏁 **Offline-capable Beta**      | 18          | installable PWA, offline study, queued sync                      |
| 🏁 **Multi-device offline Beta** | 19          | concurrent offline reconciliation proven                         |
| 🏁 **Production launch**         | 22          | hardened, monitored, deployed                                    |

Conventions: pnpm; `pnpm dev / build / typecheck / lint / test / test:e2e`;
Python data tooling stays under `scripts/`.

## Phase 6 — Study-component model & study engine (pure TS)

- **Objective:** the deterministic heart of the product — no UI.
- **Scope (in `modules/study-engine`, no React/DB imports):** component
  derivation from the release eligibility matrix (which components exist per
  entry, essential vs extended sets); lazy component materialisation API;
  the shared natural-key builder (rejects skill/shape/field/direction
  mismatches — the same module used by server later); question generation
  for all four objective types + flashcard prompts, driven by
  `(question_seed, question_generator_version, content_version,
component_key)` — fully deterministic; distractor engine (same-field,
  plausibility-ranked, normalised-uniqueness, ambiguity exclusion for
  identical surface forms); question-instance specification objects
  (`question_instance_id`, seed, generator version, `allowed_answer_refs`,
  `correct_answer_ref`, hint state, position); session state machine
  (first-attempt tracking, wrong-then-correct reinforcement re-queue, hints,
  single-step undo, timed/test modes); attempt-record creation (all fields
  from `DATA_MODEL.md` §5 incl. event-time timezone metadata); shared
  correctness logic (used optimistically now, authoritatively by the server
  in Phase 16).
- **Non-goals:** FSRS (Phase 7); UI (8–10); persistence wiring beyond thin
  adapters.
- **Prerequisites:** Phases 3, 5.
- **Expected files:** `modules/study-engine/{components.ts,generator.ts,
distractors.ts,session.ts,attempts.ts,correctness.ts}` + exhaustive tests.
- **DB changes:** none.
- **Testing checkpoint (heaviest unit-test phase):** seeded generation is
  reproducible (same seed ⇒ same question); ineligible fields never selected
  as target or distractor (property test across all entries × skills);
  duplicate-madi safety (an option set never contains a normalised-equal
  duplicate; 262/275, 297/303, 409/413 never distract each other on madi
  questions); four options, unique, answer present exactly once; bāb options
  are Arabic pairs; wrong-then-correct yields attempts
  `[incorrect(first), correct(reinforcement)]`; undo removes the last attempt
  deterministically; component derivation: māḍī and maṣdar recognition are
  distinct components, Ar→En and En→Ar distinct, bāb is one component
  regardless of prompt form; natural-key builder rejects invalid combos.
- **Acceptance criteria:** 100% of engine behaviour reachable and tested
  without a browser; deterministic regeneration proven (generate → replay
  spec ⇒ identical question).
- **Risks:** hidden nondeterminism (Date.now, Math.random) — inject clock and
  RNG; lint rule against direct use in the module.
- **Demonstrate:** test suite; a scripted session transcript printed from a
  seed.

## Phase 7 — FSRS scheduling & review events (local)

- **Objective:** spaced repetition with a causal local event chain.
- **Scope (in `modules/scheduler`, pure TS):** ts-fsrs integration (injected
  clock); rating mapping (correct→Good, hinted-correct→Hard, incorrect→Again,
  I-know→Good, I-don't-know→Again); event-creation rule (first
  scheduling-relevant attempt per component per session; recoveries create no
  event); local causal chain — events carry `parent_event_id` (head of the
  local chain or last server-accepted event), `base_server_revision`
  (guests: 0), `client_component_revision` (monotonic), event-time timezone
  fields; undo (pre-sync: delete attempt+event, restore snapshot; post-sync
  revocation deferred to Phase 16); due-item selection; learner-state
  projection (not_started/learning/mastered/needs_review with the
  three-distinct-mastery-days rule over stored `local_date_at_event`,
  Good/Easy only, initial learning excluded); mixed-revision ordering
  (due → weak → new within daily targets).
- **Non-goals:** server anything; UI.
- **Prerequisites:** Phase 6.
- **Expected files:** `modules/scheduler/{fsrs.ts,ratings.ts,events.ts,
chain.ts,states.ts,due.ts}` + tests.
- **DB changes:** Dexie v3 if fields were missed in v2.
- **Testing checkpoint:** rating mapping table-tested incl. hinted-incorrect
  → Again; wrong-then-correct ⇒ exactly one Again event + reinforcement
  attempt; sequential local chain — two and three offline reviews all
  affect FSRS in causal order with correct parent links and monotonic
  revisions; mastery: Hard never advances days; three Good reviews on one
  local date = one mastery day; initial learning date excluded; timezone
  fields recorded per event; DST/near-midnight table tests; undo restores
  exact prior state.
- **Acceptance criteria:** replaying a recorded event chain reproduces the
  card state bit-for-bit; all rules of `PRODUCT_REQUIREMENTS.md` §5
  demonstrably enforced.
- **Risks:** FSRS misuse (state vs rating confusion) — golden-file tests
  against ts-fsrs reference sequences.
- **Demonstrate:** simulated 10-day learning script showing state
  transitions and due dates.

## Phase 9 — Multiple-choice vocabulary quizzes

- **Objective:** Ar→En and En→Ar MC modes.
- **Scope:** quiz session UI (4 options, immediate-feedback + test modes,
  timed mode with per-question countdown); prompt wording treats `meaning` as
  the BASE lexical meaning ("Choose the base meaning"), never as an exact
  form translation. Ar→En: the quizzed form may stay hidden until the
  feedback, which shows the base meaning and names the form ("Form: Verbal
  noun (maṣdar)"). En→Ar: the requested form is named before answering
  ("Choose the muḍāriʿ form") — while only a base gloss exists the base
  meaning cannot identify the wanted form — and the feedback confirms it;
  wrong answers re-queued in-session; optimistic correctness from shared
  logic; session results screen (first-attempt accuracy, hinted, recovered).
- **Non-goals:** bāb/root quizzes; hints UI beyond basic (first letter +
  root + bāb hints included here if trivial, else Phase 11).
- **Prerequisites:** Phase 8.
- **Testing checkpoint:** Playwright: complete Ar→En and En→Ar sessions;
  test-mode feedback withheld until end; timed expiry counts as incorrect;
  Ar→En feedback reveals the correct form name; En→Ar names the requested
  form before answering; engine tests already cover generation — E2E asserts
  wiring only; axe scan.
- **Acceptance criteria:** §4.5 generation rules observable in UI; results
  distinguish first-attempt vs recovered.
- **Demonstrate:** one full session each direction, one test-mode session.

- **Non-goals:** offline queue durability guarantees (18); concurrent-branch

- **Prerequisites:** Phase 15.
- **Expected files:** `app/api/sync/*`, `modules/sync/*`,
  `modules/study-engine` reused server-side.

  accepted and replayed identically on repeat; timezone correction flagged;
  E2E: study signed-in on one browser context, see state on a second context
  after sync.
- **Acceptance criteria:** server component state always equals replay of
  accepted events (asserted by a replay-idempotence test); objective
  correctness is server-derived end-to-end.
- **Risks:** generator drift between client and server — one shared module,
  version pinned, covered by reconstruction tests.
- **Rollback:** sync feature flag off ⇒ app degrades to local-only.
- **Demonstrate:** two-browser cross-device session; a tampering attempt
  rejected in the audit log.

## Phase 19 — Multi-device offline reconciliation 🏁 Multi-device offline Beta

- **Objective:** correctness when the same component is reviewed on multiple
  offline devices.
- **Scope:** enable and complete the branch machinery: concurrent-branch
  detection by causal ancestry (`base_server_revision` +
  `parent_event_id`); pessimistic-rating winner, ties by canonical order;
  losing branch + scheduling descendants → `conflict_demoted`; pending-parent
  holding, TTL (~14 days), reprocessing and client chain-resubmission;
  duplicate `client_component_revision` handling; legacy 10-minute fallback
  (different devices, same component, no intervening revision, logged) only
  for invalid/missing lineage; client rebase UX ("schedule updated from your
  other devices"); mastery-day recomputation excluding demoted events;
  monitoring counters for fallback + demotion rates.
- **Prerequisites:** Phase 18.
- **Testing checkpoint (the DAG suite):** sequential 2- and 3-event
  single-device chains accepted; two-device branch resolved deterministically
  with the pessimistic rule; branch with multiple descendants — losing
  descendants demoted consistently; winner's descendants preserved;
  event-before-parent held then ingested; parent-never-arrives → recoverable
  error after TTL; cycle rejection; duplicate client revisions deterministic;
  replay determinism (shuffle arrival order ⇒ identical final state); clock
  skew cannot create false concurrency; stale `base_server_revision` alone
  never demotes; demoted events add no mastery days; E2E: two browser
  contexts offline reviewing the same card, both sync, states converge with
  one authoritative review.
- **Acceptance criteria:** Multi-device offline Beta — the §OFFLINE_AND_SYNC
  walkthrough reproduced by an automated test.
- **Risks:** the hardest phase — keep DAG logic in pure functions; property
  tests over random event orderings.
- **Demonstrate:** the two-offline-devices scenario live, plus the replay
  determinism test.
